# 🥤 CaloriSnap — 拍照识别饮品热量 & 咖啡因

> **一句话**：拍一下奶茶/咖啡杯上的标签，秒算今天喝了多少卡路里和咖啡因。

---

## 📌 项目概览

| 字段 | 内容 |
|------|------|
| **状态** | 🟡 规划中 |
| **产品形态** | 微信小程序（首选） / App |
| **目标用户** | 关注健康但又爱喝奶茶咖啡的年轻人（18-35岁） |
| **核心价值** | 用最低成本记录饮品热量 + 咖啡因摄入，无需手动翻查 |
| **启动时间** | 2026-03 |

---

## 🎯 解决什么问题

### 用户痛点

1. **查热量太麻烦** — 每次喝奶茶都想查热量，但要打开 App 搜品牌→找口味→选杯型→选加料，步骤太多
2. **信息不准确** — 网上查到的热量数据来源不一、口径不同，不同加料组合根本查不到
3. **记录难坚持** — 传统的饮食记录工具太重，专门记一杯奶茶太麻烦，最终放弃
4. **不知不觉超标** — 一周喝了几杯、摄入多少完全没概念，直到体重秤上见真章

### 现有方案的问题

| 现有方案 | 问题 |
|---------|------|
| 薄荷健康等饮食记录 App | 太重，需要手动搜索输入，专门记一杯饮品显得小题大做 |
| 百度搜索 | 数据不统一，没有考虑加料/杯型/冰量/糖度等变量 |
| 品牌官网营养表 | 不是所有品牌都公开，而且用户不会每次去查 |
| 小红书/抖音攻略 | 碎片化、非结构化，无法累积追踪 |

## 💡 产品方案

### 核心流程

```
用户拍照 → OCR识别标签 → 匹配数据库 → 计算热量+咖啡因 → 记入日志
   📸          🔍              🗄️              🧮              📊
```

### 详细用户流程

```
1. 用户买了一杯奶茶/咖啡
2. 打开小程序，拍一张杯身标签的照片
   （标签通常包含：品牌、品名、规格、加料、糖度、温度等）
3. AI 识别标签上的文字信息
4. 自动匹配饮品数据库，计算热量
   - 如果完全匹配 → 直接显示热量结果
   - 如果部分匹配 → 显示最接近的结果，让用户确认/微调
   - 如果无法匹配 → 让用户手动选择品牌+品名，同时记录新样本
5. 显示热量结果卡片：
   - 🔥 本杯热量 xxx kcal
   - 📊 相当于 x 碗米饭 / 需要跑步 x 分钟消耗
   - 📅 本周累计饮品热量
6. 自动记入饮品日志
```

### MVP 功能清单

#### P0 — 最小可用

- [ ] **拍照识别**：OCR 识别奶茶/咖啡杯身标签文字
- [ ] **热量计算**：根据品牌+品名+规格+加料计算热量
- [ ] **结果展示**：热量数字 + 直观对比（米饭/跑步等）
- [ ] **饮品日志**：自动记录每次识别，按日/周查看

#### P1 — 体验增强

- [ ] **手动输入**：识别失败时可手动选择品牌/品名
- [ ] **热门品牌库**：预置喜茶、瑞幸、星巴克、霸王茶姬、茶百道等 Top 品牌数据
- [ ] **每日/每周报告**：本周喝了几杯、总热量、趋势图
- [ ] **热量预算**：设定每日/每周饮品热量上限，超标提醒

#### P2 — 社交与增长

- [ ] **分享卡片**：生成好看的热量结果卡片，分享到朋友圈
- [ ] **饮品评分**：同价位、同类型的「性价比」排行（好喝 vs 热量）
- [ ] **打卡挑战**：连续 N 天控制饮品热量
- [ ] **社区 UGC**：用户贡献新品牌/新口味数据

## 🏗️ 技术方案（初步）

### 技术架构

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│  微信小程序   │────▶│  后端 API     │────▶│  数据库       │
│  (前端)      │     │  (Python)    │     │  (Supabase)  │
└─────────────┘     └──────┬───────┘     └──────────────┘
                           │
                    ┌──────▼───────┐
                    │  AI 能力层    │
                    │  OCR + LLM   │
                    └──────────────┘
```

### 技术选型（待确定）

| 模块 | 候选方案 | 说明 |
|------|---------|------|
| **前端** | 微信小程序 | 用户最方便，无需下载，拍照即用 |
| **后端** | Python (FastAPI) | 开发效率高，AI 生态好 |
| **数据库** | Supabase (PostgreSQL) | 免费额度够 Side Project 用 |
| **OCR** | 微信 OCR / 腾讯云 OCR / PaddleOCR | 识别杯身标签文字 |
| **AI 理解** | 大模型 API（混元/DeepSeek） | 从 OCR 文字中提取结构化信息（品牌、品名、加料等） |
| **热量数据** | 自建数据库 + 爬虫 | 各品牌官方营养信息 + 社区贡献 |
| **部署** | 腾讯云 / Vercel | 后端 API 部署 |

### 核心技术挑战

| 挑战 | 难度 | 思路 |
|------|------|------|
| **标签 OCR 准确率** | ⭐⭐⭐ | 奶茶杯标签字体小、背景花哨、有水渍——需要针对性优化预处理 |
| **信息结构化提取** | ⭐⭐ | 用 LLM 从 OCR 文字中提取品牌/品名/加料/杯型/糖度 |
| **热量数据库建设** | ⭐⭐⭐ | 初期手动整理 Top 品牌，后续靠用户 UGC + 爬虫补充 |
| **加料热量计算** | ⭐⭐ | 珍珠/椰果/芋圆等加料的热量需要单独建模 |
| **模糊匹配** | ⭐⭐ | 标签上的名称可能与数据库不完全一致，需要智能匹配 |

## 📊 饮品热量数据库设计（草案）

### 数据模型

```
brands（品牌表）
├── id, name, logo, category(奶茶/咖啡/果茶)

drinks（饮品表）
├── id, brand_id, name, category
├── base_calories（基础热量，标准杯/标准糖）
├── sizes: { 中杯: {ml, calories_factor}, 大杯: {...} }

toppings（加料表）
├── id, name, calories_per_serving
├── eg: 珍珠 100kcal, 椰果 40kcal, 芋圆 80kcal

sugar_levels（糖度表）
├── level: 正常糖/少糖/半糖/微糖/无糖
├── calories_factor: 1.0 / 0.85 / 0.7 / 0.55 / 0.4

records（用户记录表）
├── id, user_id, drink_id, toppings[], sugar_level
├── total_calories, photo_url, recognized_text
├── created_at
```

### 初期数据覆盖计划

| 优先级 | 品牌 | SKU 估算 |
|--------|------|---------|
| P0 | 瑞幸、星巴克、喜茶、霸王茶姬 | ~200 |
| P1 | 茶百道、沪上阿姨、古茗、奈雪 | ~200 |
| P2 | 蜜雪冰城、CoCo、一点点、书亦 | ~200 |
| 长期 | 用户 UGC 贡献 + 长尾品牌 | 持续增长 |
