## 🧠 记忆系统升级到 v2.0

### 升级动机
原记忆系统仅有扁平的每日日志文件（`memory/YYYY-MM-DD.md`），存在以下问题：
- 所有信息混在日志中，缺乏结构化组织
- 没有长期记忆提炼，信息随时间流失
- 无法按主题检索，只能按日期查找
- 缺乏"知道什么"和"会什么"的分离

### 新架构：三层记忆模型
参考 Mem0/MemGPT 的理论 + 认知心理学记忆分类，构建了：

| 层级 | 目录 | 功能 | 文件数 |
|------|------|------|--------|
| 语义记忆 | `semantic/` | 知识与事实（项目、技术栈、领域、人物） | 4 |
| 程序记忆 | `procedural/` | 技能与流程（工作流、模式、错题本、工具） | 4 |
| 情景记忆 | `episodic/` | 每日事件日志（保留30天，到期蒸馏） | 3 |
| 工作记忆 | `working/` | 当前任务上下文 | 1 |
| 核心索引 | 根目录 | MEMORY.md（大脑皮层）+ ARCHITECTURE.md | 2 |

### 记忆管理引擎
创建了 `memory_engine.py`，提供 6 大命令：
- `stats` — 统计仪表盘（14文件/1048行/9194字）
- `health` — 健康检查（当前评分 100/100）
- `index` — 自动生成检索索引（含关键词提取）
- `search` — 关键词搜索全部记忆
- `distill` — 蒸馏旧日志到长期记忆
- `compact` — 检查和压缩过大文件

## 🧠 记忆系统 v2.0 → v2.1 第一性原理 Review

### 批判性分析发现的 9 个问题

| # | 问题 | 严重度 | 本质 |
|---|------|--------|------|
| 1 | 启动需 3 次 read_file 才能"醒来" | 🔴 | 应该 1 次搞定 |
| 2 | MEMORY.md 是指路牌不是弹药库 | 🔴 | 读完还得再跳转 |
| 3 | 日志存 3 份（根目录+episodic），违反 SSOT | 🔴 | 同步噩梦 |
| 4 | ARCHITECTURE.md 对 AI 零价值 | 🟡 | 无消费者的产出物 |
| 5 | _index.md 华而不实 | 🟡 | 无消费者的产出物 |
| 6 | distill 命令是假自动化（只打印建议） | 🟡 | 名不副实 |
| 7 | working/current.md 存已完成列表=重复 episodic | 🟡 | 职责不清 |
| 8 | 写入规则要判断 8 个文件 | 🟠 | 复杂度导致低执行率 |
| 9 | 4个 semantic 文件过度分类（people.md 仅4行有效） | 🟠 | 为分类而分类 |

### 优化措施

| 措施 | 效果 |
|------|------|
| 重写 MEMORY.md 为自给自足弹药库 | 读完 1 个文件即可工作 |
| 合并 semantic/ 从 4→2 文件（context.md + tech_stack.md） | 消除重复，减少文件数 |
| 合并 procedural/ 从 4→2 文件（skills.md + tools.md） | 消除重复，减少文件数 |
| 删除 ARCHITECTURE.md 和 _index.md | 砍掉无消费者产出物 |
| 重建 working/current.md 为纯任务传递器 | 明确职责边界 |
| 重写 memory_engine.py：砍 3 假功能 + 新增 tree | 从 6 命令→4 命令，每个都真实有用 |
| 重写 Rule：启动 2 步 + 写入 2 步 | 从"判断 8 个文件"简化为"2 步搞定" |

### 优化前后对比
- 文件数: 14 → 9（-36%）
- 必读文件: 3 → 2（-33% 启动 tool call）
- 写入判断: 8 个文件 → 2 步
- engine 命令: 6 个（含 3 个假的）→ 4 个（全部真实可用）
